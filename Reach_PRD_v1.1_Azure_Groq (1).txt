1. Executive Summary
Reach is an AI-native Mini CRM built for D2C and retail brands to intelligently engage their shoppers. The product lets marketers describe who they want to reach in plain language, uses AI to suggest a customer segment and message, then fires personalized campaigns across WhatsApp, SMS, and Email — tracking every delivery event in real time through a two-service callback architecture.

Core thesis: AI does not replace the marketer — it amplifies them. Every AI suggestion can be reviewed, edited, and confirmed before any campaign fires.

2. Problem Statement
D2C brands with thousands of shoppers face three compounding problems:
	•	Manual segmentation is slow and brittle. Marketers spend hours building static lists that go stale the moment data changes.
	•	Generic messaging destroys trust. Batch-and-blast campaigns with no personalization lead to low open rates and high unsubscribes.
	•	Campaign performance is a black box. After sending, brands have no visibility into what actually happened — delivered, read, clicked, or ignored.

Opportunity: A single marketer with Reach should be able to — in under 5 minutes — identify a precise audience, craft a personalized message with AI help, fire the campaign, and watch delivery events arrive in real time.

5. User Stories
5.1  Data Ingestion
US-01 — CSV Upload (Priority: Must)
As a brand admin, I want to upload a CSV of customers and orders so that the CRM is seeded with realistic shopper data without manual entry.
US-02 — Customer List (Priority: Must)
As a brand admin, I want to view a searchable, paginated customer list with RFM scores so that I can quickly understand who our high-value and at-risk shoppers are.
5.2  Segmentation
US-03 — AI Intent Segmentation (Priority: Must)
As a marketer, I want to type a plain-English description of the audience I want to reach (e.g. 'women who bought twice but haven't purchased in 3 months') so that the AI resolves my intent into a segment filter and shows me the audience count instantly.
US-04 — Manual Segment Builder (Priority: Must)
As a marketer, I want to manually build a segment using filter controls (channel, recency, frequency, spend, lifecycle stage) so that I have precise control when the AI suggestion doesn't match my intent.
US-05 — Saved Segment Views (Priority: Should)
As a marketer, I want to save a segment as a named view (e.g. 'VIP Customers', 'Lapsed 90 Days') so that I can reuse audience definitions across multiple campaigns without rebuilding them.
5.3  Campaign Creation & AI Assist
US-07 — AI Message Draft (Priority: Must)
As a marketer, I want to create a campaign by picking a segment, a channel, and describing the campaign goal so that the AI (via Groq) drafts a personalized message with the customer's name, last purchase, and recency baked in.
US-08 — Edit AI Draft (Priority: Must)
As a marketer, I want to edit the AI-drafted message before sending so that I maintain full control and can adjust tone, offer, or copy before any message goes out.
5.4  Message Delivery (Channel Stub)
US-11 — Real-time Delivery Events (Priority: Must)
As a marketer, I want to fire a campaign and watch per-message delivery events arrive in real time so that I can see exactly which messages were delivered, failed, opened, or clicked as they happen.
5.5  Analytics & Insights
US-13 — Campaign Dashboard (Priority: Must)
As a marketer, I want to view a campaign-level dashboard showing sent, delivered, open rate, click rate, and failures so that I can evaluate performance and know where to improve the next campaign.
US-15 — Per-Campaign Drilldown (Priority: Must)
As a marketer, I want to drill into a single campaign and see per-message status for every shopper in the audience so that I can debug specific delivery failures or investigate high performers.

Hosting note: All services are hosted on Microsoft Azure using the $100 GitHub Student Developer Pack credit. This eliminates cold-start latency seen on hobby tiers and gives consistent uptime for the callback loop.

Key Design Decisions
	•	Two-service split: CRM and channel stub are separate Azure App Service deployments, mirroring real-world separation of campaign orchestration from delivery infrastructure.
	•	Idempotent receipts: the callback endpoint uses a (campaign_id, customer_id, event_type) unique constraint so duplicate events from retries are safe.
	•	Exponential backoff in stub: failed callback retries use 1s → 2s → 4s → dead letter, making failure handling explicit and observable.
	•	Groq over Anthropic: Groq's free tier offers ~500 RPD at zero cost. llama-3.3-70b-versatile provides excellent instruction-following for JSON-structured segment filter generation. At production scale, this swaps to a managed LLM gateway.
	•	Scale tradeoff stated: the current design handles ~10K messages/campaign. At 1M+, the send pipeline would move to Azure Service Bus with consumer groups per channel.

11. Open Questions & Tradeoffs
Question / Tradeoff
Decision
Which AI provider?
Groq (free tier) with llama-3.3-70b-versatile. Near-zero latency, no cost. Swappable to GPT-4o or Gemini at production scale via an abstraction layer.
How much autonomy should the AI have?
AI suggests; marketer confirms. No campaign fires without explicit approval. Trust is built through transparency.
Polling vs. WebSocket for live stats?
WebSocket (socket.io) for real-time delivery events on the campaign drilldown page. Polling for summary dashboard refresh.
How to handle the callback if CRM is down?
Stub retries with exponential backoff up to 5 attempts, then dead-letters the event. Logged for manual replay.
Azure vs Vercel/Railway?
Azure used throughout to maximize the $100 GitHub Student credit. Azure Static Web Apps for Next.js frontend + App Service for the stub + PostgreSQL Flexible Server for DB — all under one subscription.
At scale: what breaks first?
The synchronous send loop. At 100K+ recipients, this moves to Azure Service Bus with worker processes per channel.
